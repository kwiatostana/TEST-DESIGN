
# План тестирования
**Система:** Сервис А + Сервис B + Брокер сообщений (RabbitMQ)
 
## 1. Допущения (Assumptions)
 
- **Коды ответов:** Сервис А возвращает HTTP 202 Accepted, не дожидаясь ответа от Сервиса B.
- **Идемпотентность:** повторная отправка одного и того же запроса не дублирует команду в брокере.
- **Текущее состояние услуги:** запрос на уже текущее состояние (например, подключить уже подключённую) не отправляет команду в брокер.
- **Таймауты:** при потере AMQP-сообщения Сервис А ждёт ответа не бесконечно, а по таймауту.
- **Логирование:** Сервис А логирует ошибки при таймауте, недоступности RabbitMQ и невалидном ответе от Сервиса B.
 
## 2. Объект тестирования
Взаимодействие двух сервисов через брокер сообщений:
- **Сервис А** - принимает REST-запросы, управляет очерёдностью, отправляет AMQP-команды.
- **Сервис B** - получает команды, выполняет операции, возвращает результат по AMQP.
- **RabbitMQ** - брокер сообщений между сервисами.
 
## 3. Виды тестирования
- Функциональное тестирование
- Интеграционное тестирование
- Негативное тестирование
- Тестирование обработки отказов
 
## 4. Тестовое окружение и предусловия
- RabbitMQ поднят, очереди для команд и ответов созданы.
- Для тестирования отказов допускаются моки или ручная остановка процессов.
- Перед каждым тестом окружение сброшено в начальное состояние.
 
## 5. Тест-кейсы
 
### REST API Сервиса А
- Запросы на подключение, отключение и блокировку принимаются с HTTP 202, ответ Сервиса B не ожидается.
- Запрос с невалидными данными (несуществующий serviceId, отсутствие обязательных параметров) возвращает ошибку, AMQP-сообщение не отправляется.
- Повторный одинаковый запрос не порождает вторую команду в брокере.
- Запрос на уже текущее состояние услуги возвращает ответ, команда в брокер не уходит.
 
### Логика изменения статуса услуги
- Статус не меняется сразу при получении REST-запроса.
- Статус меняется только после положительного AMQP-ответа от Сервиса B.
- При отрицательном ответе статус остаётся прежним.
 
### Передача AMQP-команд и упорядочивание
- Во время ожидания ответа от Сервиса B Сервис А продолжает принимать новые REST-запросы и добавлять их в буфер.
- AMQP-команда соответствует типу REST-запроса.
- При нескольких быстрых запросах подряд в брокер уходит только первая команда, остальные ждут в буфере Сервиса А.
- После положительного ответа Сервис А отправляет следующую команду из буфера.
- После отрицательного ответа Сервис А не блокируется и тоже отправляет следующую команду.
 
### Нестандартные сценарии (Edge cases)
- Сервис B недоступен - Сервис А продолжает принимать запросы, первая команда ждёт в RabbitMQ, остальные в буфере. Статусы не меняются.
- Сервис B восстановился - накопленные команды уходят по порядку, статусы обновляются.
- Таймаут ответа от Сервиса B - запрос помечается ошибочным, статус не меняется, следующая команда из буфера отправляется.
- RabbitMQ недоступен - Сервис А возвращает ошибку на REST-запрос, команда не фиксируется.
- Сервис B вернул невалидный ответ (неизвестный статус, битый формат) - Сервис А его игнорирует, статус не меняется, очередь идёт дальше.
