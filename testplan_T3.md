
# План тестирования
**REST API:** `GET /products/{productId}/cost?authToken=&count=&start=`

---

## 1. Объект тестирования

```
GET /products/{productId}/cost?authToken=&count=&start=
```

---

## 2. Цель

Проверить корректность работы эндпоинта.

---

## 3. Область тестирования (Scope)

- Параметр `authToken`: валидация, граничные значения, спецсимволы.
- Параметр `productId`: существующий / несуществующий / невалидный формат / нулевое и отрицательное значение.
- Параметр `count`: граничные значения, невалидные типы данных.
- Параметр `start`: граничные значения, невалидные типы данных.
- Бизнес-логика: расчёт стоимости нарастающим итогом для различных комбинаций `start` и `count`.
- Поведение при `count` > 5: расчёт выполняется партиями по 5 экземпляров.
- HTTP-коды ответа: 200, 401, 404, 500.
- Формат и содержимое тела ответа (JSON-структура).

---

## 4. Техники тест-дизайна

| Техника | Где применяется |
|---|---|
| Эквивалентное разбиение | Все параметры: валидные / невалидные значения |
| Граничные значения (BVA) | `authToken`: 15, 16, 17 символов; `count`: 0, 1, 5, 6; `start`: -1, 0, 1 (границы нижней части), 4, 5 (границы верхней части) |
| Таблица принятия решений | Приоритет проверок `authToken` и `productId` |
| Предугадывание ошибок | Нестандартные случаи: пустые значения, отрицательные числа, дроби |

---

## 5. Допустимые значения параметров

| Параметр | Допустимые значения |
|---|---|
| `count` | `1`, `2`, `3`, `4`, `5` (и более, кратно партиям) |
| `start` | `0`, `1`, `2`, `3`, `4` |
| `authToken` | 16 символов, содержащих латинские буквы (a-z, A-Z) и цифры (0-9), обязательно наличие обоих типов |
| `productId` | Числовой идентификатор существующего продукта |

---

| `count` - дробное число | `1.5` | Невалидный тип данных |
| `count` - строка | `abc` | Невалидный тип данных |

| `count` - параметр отсутствует | *(параметр не передаётся)* | Полное отсутствие параметра в запросе |
| `start` ниже минимума | `-1` | Ниже нижней границы (BVA) |
| `start` выше верхней границы | `5` | Выше верхней границы (BVA) |
| `start` - строка | `abc` | Невалидный тип данных |
| `start` - пустое значение | *(пусто)* | Невалидный тип данных |
| `start` - параметр отсутствует | *(параметр не передаётся)* | Полное отсутствие параметра в запросе |

## 8. Тест-кейсы

> **Общие предусловия (если не указано иное):**
> 1. Тестовое окружение доступно по адресу `{BASE_URL}`
> 2. В системе существует `productId=1`
> 3. Валидный `authToken`: `Ab1Cd2Ef3Gh4Ij5K`
> 4. Инструмент для выполнения запросов: Postman (или curl). GET-запрос отправляется на `{BASE_URL}`, ответ проверяется в теле и заголовках ответа.

### 8.2. Бизнес-логика, расчёт стоимости нарастающим итогом

Цель: проверить корректность значения `cost` при различных комбинациях `start` и `count`.

**Общий шаблон:**
- Предусловия: общие
- Шаги: Отправить GET-запрос на `{BASE_URL}/products/1/cost?authToken=Ab1Cd2Ef3Gh4Ij5K&count={count}&start={start}`
- Ожидаемый результат: HTTP-код **200 OK**. Тело ответа: `{"cost": {ожидаемая стоимость}}`. Тип поля `cost` число.

| ID | `start` | `count` | Расчёт | Ожидаемый `cost` |
|---|---|---|---|---|
| TC_BIZ_001 | `0` | `5` | 30+25+20+15+10 | `100` |
| TC_BIZ_002 | `4` | `1` | 10 | `10` |
| TC_BIZ_003 | `1` | `3` | 25+20+15 | `60` |

---

### 8.3. Тестирование партий (count > 5)

Цель: проверить корректность расчёта стоимости при `count` > 5. Согласно ТЗ если запрашивается более 5 экземпляров, стоимость считается партиями по 5: цена сбрасывается к началу таблицы для каждой новой партии.

**Общий шаблон:**
- Предусловия: общие
- Шаги: Отправить GET-запрос на `{BASE_URL}/products/1/cost?authToken=Ab1Cd2Ef3Gh4Ij5K&count={count}&start={start}`
- Ожидаемый результат: HTTP-код **200 OK**. Тело ответа: `{"cost": {ожидаемая стоимость}}`. Тип поля `cost` число.

| ID | `start` | `count` | Расчёт | Ожидаемый `cost` | Обоснование выбора значений |
|---|---|---|---|---|---|
| TC_BAT_001 | `0` | `6` | (30+25+20+15+10)+30 | `130` | count=6: минимальный выход за границу партии (BVA, верхняя граница +1) |
| TC_BAT_002 | `0` | `10` | (30+25+20+15+10)+(30+25+20+15+10) | `200` | count=10, две полные партии |
| TC_BAT_003 | `4` | `2` | 10+30 | `40` | start=4: последняя позиция партии; count=2 перекрывает границу, вторая единица из начала новой партии |
