## Баг-репорт 2 (Требование: неточная формулировка)

| Атрибут | Значение |
| :--- | :--- |
| **Короткое описание** | Требование к поиску файлов в корзине сформулировано неоднозначно |
| **Проект** | Яндекс.Диск |
| **Компонент** | Документация / Требования - Поиск по файлам и папкам |
| **Тип** | Баг требований |
| **Severity** | S3 Major |
| **Priority** | Medium |
| **Статус** | Новая |
| **Автор** | kwiatostana |
| **Описание проблемы** | В требованиях указано: *"Если файл находится в корзине - он так же может быть найден поиском."* Формулировка "может быть найден" допускает две трактовки: <br> 1) файл находится автоматически общим поиском; <br> 2) файл можно найти только через ручное переключение фильтра на "Корзина" |
| **Ожидаемый результат** | Требование однозначно описывает поведение: указано, происходит поиск по корзине автоматически или через фильтр |
| **Фактический результат** | Требование допускает двойное толкование, поведение системы невозможно однозначно верифицировать |
| **Комментарий** | Рекомендуется переформулировать требование, например: *"Файл из корзины отображается в результатах общего поиска без дополнительных действий"* |

## Баг-репорт 5 (Требование: незадокументированное поле details)

| Атрибут | Значение |
| :--- | :--- |
| **Короткое описание** | В теле ответа при ошибке присутствует незадокументированное поле `details` |
| **Проект** | Яндекс.Диск API |
| **Компонент** | Документация / Требования - PUT /v1/disk/resources |
| **Тип** | Баг требований |
| **Severity** | S4 Minor |
| **Priority** | Low |
| **Статус** | Новая |
| **Автор** | kwiatostana |
| **Описание проблемы** | В требованиях тело ответа при ошибке описано как: `{ "message": "string", "description": "string", "error": "string" }`. Фактически сервер опционально возвращает дополнительное поле `"details": object (optional)`, которое в требованиях не упомянуто |
| **Ожидаемый результат** | Требование содержит описание всех полей тела ответа, включая `details` |
| **Фактический результат** | Поле `details` в требованиях отсутствует |
| **Комментарий** | Рекомендуется добавить в требования: `"details": object (optional) - дополнительные детали ошибки` |

| **Тип** | Баг требований |

## Список некорректных требований

|---|---|---|---|
| 1 | "Если файл находится в корзине - он так же может быть найден поиском" | Ясность | "Может быть найден" допускает две трактовки: файл отображается автоматически в общем поиске или только через ручное переключение фильтра на "Корзина". Требование должно иметь единственную интерпретацию. |
| 2 | "В случае успешного выполнения запроса получен ответ 200" | Корректность | Сервер возвращает 201 Created. Требование содержит неверный код успешного ответа, не соответствующий REST-стандарту, официальной документации, реальному поведению сервера.  |
| 3 | "При попытке создания уже по существующему пути - ответ 400" | Корректность | Сервер возвращает 409 Conflict. Требование содержит некорректный код ошибки, не соответствующий REST-стандарту, официальной документации, реальному поведению сервера. |
| 4 | Тело ответа при ошибке: { "message": "string", "description": "string", "error": "string" }
 | Полнота | Сервер опционально возвращает дополнительное поле {"details": object}, которое в требованиях не описано. |
| 5 | Authorization: [token] | Ясность, Полнота| Не указан используемый префикс OAuth.  Оба варианта технически принимаются сервером, но требование должно явно фиксировать корректный формат чтобы избежать разночтения. |
